1.2 SOLID (5 Principios de Diseño)

Son 5 reglas que ayudan a que el código sea más fácil de mantener, extender y entender. Cada letra corresponde a un principio.

S — Single Responsibility (Responsabilidad Única)

Cada clase tiene UNA sola razón para cambiar.

En nuestra API:

Carpeta Responsabilidad
controllers/ Solo recibe peticiones HTTP
servicios/ Solo lógica de negocio (validar, normalizar)
repositorios/ Solo acceso a datos (SQL)

Si mañana cambian las reglas de validación, solo tocas servicios/. Si cambia la sintaxis SQL de PostgreSQL, solo tocas repositorios/. El controller no se entera de nada.

O — Open/Closed (Abierto/Cerrado)

Abierto para extensión, cerrado para modificación.

Si el día de mañana necesitas agregar soporte para Oracle, solo creas una clase nueva y la registras en la fábrica. No tocas nada de lo que ya funciona:

Agregar Oracle = crear RepositorioLecturaOracle
                 + 1 línea en fabrica_repositorios.py
                 + 0 cambios en controllers/ o servicios/

L — Liskov Substitution (Sustitución de Liskov)

Cualquier repositorio (PostgreSQL, SQL Server, MySQL) puede reemplazar a otro sin que el servicio se entere.

Prueba: Cambiar DB_PROVIDER=postgres por sqlserver en .env
        ¿Funciona sin tocar código? → Sí → Cumple Liskov

Ejemplo concreto: hoy trabajamos con bdfacturas_postgres_local. Si cambiamos el .env para apuntar a bdfacturas_sqlserver_local, la API sigue funcionando igual porque todos los repositorios cumplen el mismo contrato.

I — Interface Segregation (Segregación de Interfaces)

Interfaces pequeñas y específicas, no una interfaz gigante que haga todo.

Interfaz (Protocol) Responsabilidad
IRepositorioLecturaTabla CRUD (obtener, crear, actualizar, eliminar)
IProveedorConexion Solo cadena de conexión
IServicioCrud Solo lógica de negocio CRUD

Cada interfaz (en Python se llaman Protocol) define solo lo que necesita. Nadie implementa métodos que no usa.

D — Dependency Inversion (Inversión de Dependencias)

Las clases dependen de abstracciones (Protocols), no de implementaciones concretas.

ServicioCrud recibe: IRepositorioLecturaTabla       (Protocol / abstracción)
             NO:     RepositorioLecturaPostgreSQL    (clase concreta)

Así, el servicio funciona con cualquier repositorio que cumpla el contrato, sin importar si es PostgreSQL, SQL Server o MySQL.

Resumen SOLID en nuestra API

Principio Cómo se aplica
S — Responsabilidad Única Controller, Servicio y Repositorio separados
O — Abierto/Cerrado Nueva BD = nueva clase, sin tocar las existentes
L — Sustitución Liskov Cambiar proveedor en .env, todo sigue funcionando
I — Segregación Interfaces Protocols pequeños y específicos
D — Inversión Dependencias Servicios reciben abstracciones, no clases concretas

1.3 Arquitectura en 3 Capas

La API se divide en tres capas. Cada una tiene una responsabilidad clara y solo conoce a la capa que está inmediatamente debajo.

CLIENTE (Frontend / Swagger)
↓ HTTP
CAPA 1: PRESENTACIÓN (Controllers)
• Recibe peticiones HTTP
• Retorna respuestas JSON
• NO contiene lógica de negocio
Archivos: controllers/*.py
↓ Llama a
CAPA 2: NEGOCIO (Servicios)
• Validaciones de negocio
• Normalización de datos
• Orquesta llamadas al repositorio
Archivos: servicios/*.py
↓ Llama a
CAPA 3: DATOS (Repositorios)
• Consultas SQL
• Conexión a base de datos
• NO contiene lógica de negocio
Archivos: repositorios/*.py
↓ Consulta
BASE DE DATOS
SQL Server | PostgreSQL | MySQL
(Ejemplo: bdfacturas_postgres_local)

Regla clave: Cada capa solo conoce a la capa inmediatamente inferior.

Capa Conoce a NO conoce a
Controller Servicio Repositorio, BD
Servicio Repositorio (vía abstracción) Controller, BD directa
Repositorio BD (PostgreSQL, SQL Server, etc.) Controller, Servicio

1.5 Patrones de Diseño Usados

1. Repository Pattern (Patrón Repositorio)

Encapsula el acceso a datos. El servicio pide datos sin saber qué SQL se ejecuta ni qué motor de BD hay debajo.

Ejemplo: el servicio dice “dame los productos”. Dependiendo del motor, el repositorio ejecuta:

Motor SQL que ejecuta
PostgreSQL SELECT * FROM "public"."producto" LIMIT 1000
SQL Server SELECT TOP (1000) * FROM [dbo].[producto]
MySQL / MariaDB SELECT * FROM `producto` LIMIT 1000

El servicio no se entera de la diferencia. Solo pide datos y los recibe.

2. Service Layer (Capa de Servicio)

Capa intermedia con la lógica de negocio. El controller delega al servicio; el servicio delega al repositorio.

Ejemplo: antes de crear un registro, el servicio valida que los datos no estén vacíos y puede encriptar contraseñas con BCrypt.

3. Factory Pattern (Patrón Fábrica)

fabrica_repositorios.py decide qué repositorio crear según DB_PROVIDER en el .env. Un solo lugar centralizado.

Flujo completo (ejemplo con PostgreSQL y bdfacturas_postgres_local):

GET /api/producto
↓
Controller: “Necesito listar productos”
↓
Servicio: “Valido parámetros, delego al repositorio”
↓
Fábrica: “DB_PROVIDER=postgres → creo RepositorioLecturaPostgreSQL”
↓
Repositorio: SELECT * FROM "public"."producto" LIMIT 1000
(contra bdfacturas_postgres_local)
↓
JSON → Cliente

Resultado: Cambiar de base de datos = cambiar 1 línea en .env.

Cambio en .env Base de datos que usa
DB_PROVIDER=postgres bdfacturas_postgres_local
DB_PROVIDER=sqlserver bdfacturas_sqlserver_local
DB_PROVIDER=mysql bdfacturas_mysql_local
DB_PROVIDER=mariadb bdfacturas_mariadb_local
